# Orion Chat 超长 Paste 外置存储设计

**日期**: 2026-03-08
**状态**: Approved
**目标**: 将超长 paste 正文从消息文本中剥离出来，改为文件系统持久化 + 数据库引用元数据的模式，同时保持搜索、导出、自动压缩、编辑和模型上下文行为与当前用户体验一致。

**Implementation Note:** The first implementation batch keeps the existing frontend inline-paste payload format and externalizes legacy `<<paste:...>>...<</paste>>` markers on the Rust side before persistence. This avoids widening the Tauri send/update payloads in the same batch while still moving stored oversized paste bodies out of the messages table.

## 背景

当前项目对超长粘贴的处理方式是：
- 前端在输入区检测到长文本后，把它包裹成 `<<paste:LENGTH>>...正文...<</paste>>`
- 这个完整标记串直接写入消息内容
- 消息展示时再从正文里解析出 paste 片段并显示为标签

现状的问题是：
- 超长 paste 真实正文仍然保存在消息表里，导致数据库体积和热数据膨胀
- 长 paste 会跟着消息分页一起进入前端内存，增加多会话占用
- 编辑、压缩、导出、搜索都与这段大文本强耦合在同一字段里
- 后续要做虚拟列表时，消息内容的真实大小和展开行为也会变得更难控制

本轮需求边界已经确认：
- 搜索必须能搜到 paste 正文
- 导出必须把 paste 正文还原出来
- 自动压缩时必须把 paste 正文展开给模型
- 存储方案采用“文件系统存正文，数据库存引用元数据”
- 旧消息中已经存在的内嵌 `<<paste:...>>...<</paste>>` 数据必须继续兼容

## 方案对比

### 方案 A：文件系统存正文，数据库存引用元数据（选定）

**做法**：
- 超长 paste 正文写入应用数据目录下的独立文本文件
- 数据库新增 `paste_blobs` 表记录 paste 与消息的关联关系
- 消息正文中只保存轻量引用标记，而不再保存 paste 的真实内容
- 读取消息时按需解析引用；需要完整内容时再从文件系统读取

**优点**：
- 真正降低消息表热数据体积
- 前端默认不必把 paste 正文一并加载到内存
- 文件生命周期、缓存清理、统计空间占用都更自然

**缺点**：
- 需要处理文件与数据库记录的一致性
- 删除消息/会话时需要同步删除关联文件

### 方案 B：单独建 SQLite 表保存 paste 正文

**不选原因**：
- 虽然消息表会变轻，但超长正文仍在数据库里
- 对 WebView 和数据库体积的压力缓解有限
- 不如文件系统方案适合“很大但低频访问”的正文数据

### 方案 C：只在前端显示层外置，后端继续存完整正文

**不选原因**：
- 只能减少一部分渲染成本，不能解决持久层和多会话内存问题
- 与本次“外置存储”的目标不符

## 存储模型设计

### 新增数据库表 `paste_blobs`

建议新增一张独立表，字段包括：
- `id`：主键，UUID
- `conversation_id`：所属会话
- `message_id`：所属消息
- `char_count`：原文长度
- `file_path`：相对于应用数据目录的文件路径
- `created_at`：创建时间

如果后续需要更细粒度扩展，也可预留：
- `mime_type`
- `encoding`
- `hash`

本轮先保持最小字段集合，不做过度设计。

### 文件存储位置

建议存放在应用数据目录下，例如：
- `<app_data>/pastes/<paste_id>.txt`

要求：
- 统一 UTF-8 文本存储
- 文件名使用 UUID，避免路径冲突和注入问题
- 路径尽量用相对路径写入 DB，避免用户目录迁移后失效

## 消息正文格式设计

### 新格式：仅存引用

新消息中的 paste 标记从旧格式：
- `<<paste:LENGTH>>真实正文<</paste>>`

改为新格式：
- `<<paste-ref:PASTE_ID:LENGTH>>`

其中：
- `PASTE_ID` 对应 `paste_blobs.id`
- `LENGTH` 继续保留，便于前端直接显示标签文案而不必先查文件

### 双格式兼容

本轮不强制做历史迁移脚本。

系统需要同时支持：
- **旧格式**：消息正文里仍然直接包含 paste 正文
- **新格式**：消息正文里只包含 `paste-ref`

兼容策略：
- 展示层同时识别两种格式
- 搜索层同时覆盖两种格式
- 导出层同时把两种格式都还原成最终正文
- 模型上下文构建层同时能把两种格式都展开为真实文本

## 写入链路设计

### 输入区

当前 `InputArea.svelte` 已经能识别长 paste，并把它编码进消息字符串。

本轮调整为：
- 前端仍识别超长 paste
- 但不再把真实正文塞进消息最终内容字符串
- 改为把 paste 组织成结构化附件列表发送给后端

发送接口新增一个可选参数，例如：
- `pasteBlobs: Array<{ clientToken: string; text: string; charCount: number }>`

同时消息正文中的对应位置先放前端占位符，例如：
- `<<paste-upload:CLIENT_TOKEN:LENGTH>>`

后端在落库前将其替换成正式的 `<<paste-ref:PASTE_ID:LENGTH>>`

### 后端发送命令

`send_message` / 相关编辑重发链路需要支持接收 paste 附件参数：
- 为每个 paste 生成正式 `paste_id`
- 把正文写入文件
- 往 `paste_blobs` 插入记录
- 将消息正文中的上传占位符替换为正式引用标记
- 最终再创建消息记录

## 读取与展示设计

### 列表展示

聊天列表默认只显示 paste 标签：
- 对旧格式：沿用当前解析逻辑
- 对新格式：解析 `paste-ref` 并显示 `[N 字符]`

默认不自动读取文件正文。

### 点击查看正文

用户点击 paste 标签时：
- 若是旧格式，直接用消息中内嵌正文展示
- 若是新格式，通过新命令按 `paste_id` 读取对应文件正文
- 读取后在弹层里显示内容

### 编辑消息

消息进入编辑态时：
- 旧格式：沿用当前逻辑
- 新格式：需要把 `paste-ref` 对应的真实正文按需读回，并重组为可编辑态中的 paste 标签
- 提交编辑时，再重新走“正文外置 + 引用替换”的写入流程

## 搜索设计

搜索必须覆盖 paste 正文，因此不能只搜索消息表正文。

### 推荐方案

新增 `paste_blobs_fts` 或等效全文索引，使搜索来源包括两部分：
- 消息正文（现有 `messages_fts`）
- paste 正文（新的 paste FTS）

搜索行为：
- 如果 query 命中消息正文，返回对应消息
- 如果 query 命中 paste 正文，也返回对应消息
- 返回结果仍保持 `Message[]` 为主，不额外暴露 paste 命中细节

这样可以保持现有前端搜索调用方式不变。

## 导出设计

Markdown / JSON 导出时，需要对新旧格式都做正文还原：
- 旧格式：直接提取原文
- 新格式：读取 `paste-ref` 对应文件，把内容展开回导出文本

要求：
- 导出的 Markdown / JSON 对用户来说仍然像“完整原文对话”
- 不把内部 `paste-ref` 暴露给导出结果

## 自动压缩与模型上下文设计

### 自动压缩

构造压缩请求时：
- 若消息中存在旧格式 paste，按现有逻辑展开
- 若消息中存在新格式 `paste-ref`，需要先读取文件正文，再拼接成真实文本发送给模型

### 正常聊天请求

不仅自动压缩，普通聊天上下文构建也应采用同一套展开逻辑：
- 发送给模型之前，把消息中的 paste 引用统一还原为真实正文
- 数据库存的仍是引用，不改变持久化语义

建议把这套逻辑抽成统一 helper，避免聊天发送、重发、压缩、导出各写一套。

## 删除与清理设计

### 消息删除

当消息被硬删除或软删除时：
- 首批建议至少在“硬删除整段会话/消息”场景清理关联 paste 文件
- 对软删除是否立即清理，可先保守处理为“不立即删除文件”，以便恢复消息时仍可用

### 会话删除

删除整个会话时：
- 删除关联 `paste_blobs` 记录
- 删除对应正文文件

### 孤儿文件

首批可以暂不做自动 GC，但建议：
- 至少提供一个内部清理 helper
- 后续可挂到“清理缓存”或“修复数据”工具项里

## 兼容性与迁移策略

### 旧数据兼容

必须支持旧消息继续：
- 展示
- 搜索
- 导出
- 自动压缩
- 重发/编辑

### 历史迁移

本轮不做批量迁移脚本。

原因：
- 风险高
- 需要扫描全部历史消息并重写内容
- 容易把当前优化任务膨胀成数据迁移工程

可在未来单独做：
- 后台按需迁移
- 或仅对用户主动编辑/重发过的消息完成“新格式重写”

## 风险与缓解

### 风险 1：文件与数据库记录不一致

**缓解**：
- 后端统一负责写文件 + 写 DB + 写消息内容替换
- 尽量把“生成引用”收敛在一个命令层 helper 中

### 风险 2：搜索实现复杂度上升

**缓解**：
- 先保持搜索返回 `Message[]` 不变
- 在后端内部增加 paste FTS 和去重逻辑，不把复杂性泄露到前端

### 风险 3：编辑链路容易漏掉 paste-ref 展开

**缓解**：
- 提前抽象“消息正文 <-> 展示片段 <-> 存储引用”的统一转换函数
- 不在 `InputArea` 和 `MessageBubble` 各写一套非对称逻辑

## 与虚拟列表的关系

超长 paste 外置存储应当先于虚拟列表落地：
- 它先稳定消息数据形态
- 让单条消息的热载荷显著下降
- 后续做虚拟列表时，动态高度和内容展开场景会更可控

虚拟列表建议作为下一份单独设计处理，而不是与本轮数据层改动一起混做。

## 验证策略

### 后端

至少覆盖：
- 新 paste 文件写入与 `paste_blobs` 记录创建
- `paste-ref` 展开为真实正文
- 搜索能命中 paste 正文
- 导出能还原 paste 正文
- 自动压缩/聊天请求构建时能展开新格式 paste
- 旧格式消息继续兼容

### 前端

- `pnpm run check`
- 手动验证：
  - 发送超长 paste 后聊天中只显示标签
  - 点击标签可查看正文
  - 编辑后仍能保留 paste 标签语义
  - 搜索 paste 正文可以定位到消息
  - 导出结果包含真实正文

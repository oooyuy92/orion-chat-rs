# Orion Chat - Worker-First Agent Chat Design

**日期**: 2026-03-20
**状态**: 已讨论确认
**作者**: OpenCode + 用户协作

---

## 一句话定义

这是一个默认进入 Agent 模式的桌面 Chat；每条会话只围绕一个持续任务线程展开，AI 先给出简短计划，再在对话中执行工作、按需读取工作区和写脚本，过程可见但代码默认隐藏，重复操作会自然沉淀成 skill。

---

## 背景与动机

当前仓库已经具备桌面 Chat、Agent 工具执行、工作目录、MCP 接入口和工具授权等基础能力，但产品形态仍更接近「带 Agent 能力的聊天客户端」。

目标不是把产品做成编程工具，也不是把用户带进显式的自动化搭建器，而是做成一个面向真实工作需求的 worker：

- 用户主要通过聊天表达目标
- AI 能理解需求并拆解任务
- 必要时在后台写脚本完成资料处理或自动化
- 用户默认看不到代码细节，但能看懂系统正在做什么
- 重复流程能被沉淀为可复用的 skill

---

## 目标

- 保持 `conversation/message` 为产品顶层结构，不把首页改造成任务看板或 IDE
- 默认新会话进入 `agent chat`
- 用户可以随时关闭 Agent，退回普通聊天
- 每条会话只围绕一个持续任务线程展开，不鼓励在同一会话内混入明显不同的任务
- AI 接到明确需求后，先给出 3-5 步简短计划，再进入执行态
- 执行态在聊天流中完成，通过自然语言、步骤状态和动画让用户感知工作过程
- 资料既可以直接通过聊天输入，也可以在需要时绑定工作区
- 需要脚本时，AI 可以在后台生成并执行脚本，但代码默认隐藏
- 当系统识别到重复工作流时，在聊天中自然建议升级为 skill
- 用户后续可以通过自然语言让 AI 修改已有 skill，以适应同类工作流的微小变化

## 非目标

- 不把产品做成显式的 workflow builder
- 不把代码编辑器、终端和 diff 作为主界面中心
- 不追求「适应所有任务」的万能入口，第一阶段聚焦资料处理和脚本型自动化
- 不采用「每次脚本都自由选择语言」的多语言执行方案

---

## 已讨论的产品选择

### 顶层形态

已选：

- 首页主角仍然是 `会话列表 + 当前对话`
- 默认是 `agent chat`
- 用户可以关闭 Agent，进入普通聊天

未选：

- 任务工作台优先
- 显式 automation builder

### 会话边界

已选：

- 一条会话对应一个持续任务线程
- 如果用户在同一会话里切换到明显不同的任务，AI 应建议用户新开会话

未选：

- 把同一会话当成长期万能窗口，混合多个互不相关任务

### 执行反馈

已选：

- 自然语言解释 + 结构化步骤状态 + 动画

未选：

- 纯黑盒自动执行
- 纯流程图式执行器

### 资料输入方式

已选：

- 资料可直接进入聊天
- 也可以在需要时绑定工作区

未选：

- 强制用户先选工作区再开始

### skill 触发方式

已选：

- 由 AI 在聊天里自然发起建议

未选：

- 突然切换到复杂管理页或 builder

### 自治与风险

已选：

- 三档自治：每次询问 / 允许修改 / 完全自主
- 即使在完全自主模式下，高风险动作仍然必须严格保护

---

## 推荐产品方案

推荐采用 `Worker-first Chat`：

- 产品的外层感受是 Chat
- 产品的内层骨架是一个可视化 worker
- 用户以为自己一直在聊天，但系统实际上在推进一个任务线程

相比之下：

- 单纯强化 Chat，只会得到一个偶尔会跑脚本的聊天工具
- 一开始就做显式工作流平台，会让第一版过重，偏离当前仓库结构和目标用户心智

---

## 核心对象模型

### Conversation

`Conversation` 仍是顶层容器，但它的产品语义不再只是消息历史，而是一个单任务工作线程。

它应该天然承载：

- 当前任务主题
- Agent 是否开启
- 会话级工作区
- 本会话产生的交付物
- 本会话沉淀出的重复流程线索

### Message

`Message` 仍保留聊天主界面所需的自然对话感，但不能再只是文本容器。

消息需要逐步承载：

- 简短计划
- 执行中自然语言说明
- 步骤状态引用
- 结果块引用
- skill 建议

### Operation

`Operation` 是完成某次具体工作的一次执行。

它不是首页主对象，但在系统内部应成为一等概念，用来表示：

- 这次任务的目标
- 使用了哪些输入材料
- 是否生成了脚本
- 执行过程和结果如何
- 是否值得升级为 skill

### Artifact

`Artifact` 是交付物，必须从纯文本消息里独立出来。

可以包括：

- 整理后的文本结果
- 导出的文件
- 脚本输出
- 处理后的目录结果

对用户来说，artifact 的价值是「这个结果可以继续被引用」，而不只是「AI 说它做完了」。

### Skill

`Skill` 是被确认可复用后的工作能力，不等于单个脚本文件。

它至少需要包含：

- 名称
- 用途说明
- 适用任务范围
- 输入输出约束
- 对应脚本路径
- 版本信息

## 首页与会话结构

### 首页

首页继续采用：

- 左侧会话列表
- 中间聊天流

不把首页变成看板，也不把执行器做成另一个主页面。

### 新会话默认态

默认新会话进入 `agent chat`。

输入区附近保留轻量 `Agent` 开关：

- 开启：worker-first chat
- 关闭：普通聊天

### 会话标题与摘要

会话标题不应只是聊天主题，而应逐步贴近工作对象，例如：

- 竞品资料整理
- 客户数据清洗
- 周报自动汇总

这样左侧列表自然成为工作线程列表。

### 单任务约束

会话默认只服务一个持续任务线程。

如果用户明显切换到新的目标，AI 应建议新开会话，而不是继续把上下文和交付物混在一起。

---

## 执行态设计

### 开始执行前

当用户提出明确任务时，AI 不应直接长篇回答，而应先给出一个 3-5 步简短计划。

目标是：

- 让用户知道系统理解了什么
- 让用户知道接下来要做什么
- 为后续自治建立信任

### 执行态仍在聊天流中

执行不跳出聊天页。

聊天流中的关键消息旁应挂接步骤状态卡片，用来展示：

- 当前步骤名称
- 当前状态
- 轻量动画
- 是否需要用户确认

### 双层反馈

执行态同时提供两层信息：

- 自然语言层：AI 用人话解释正在做什么
- 结构化层：系统用步骤状态明确告诉用户正在读取资料、分析结构、生成脚本、执行脚本或生成交付物

### 代码默认隐藏

用户默认看到的是动作和进度，而不是脚本正文、终端输出或 diff。

对高级用户，可以保留查看路径、运行记录和脚本细节的入口，但不是主流程。

### 风险状态切换

执行态必须支持三档自治：

- 每次询问
- 允许修改
- 完全自主

同时保留硬性高风险拦截：

- 删除
- 覆盖
- 危险 shell
- 对用户文件产生不可逆副作用的动作

当遇到高风险动作时，状态应明确从「执行中」切到「等待确认」。

---

## 资料输入与工作区

### 两条等价入口

资料输入有两条入口：

- 直接通过聊天粘贴、拖拽、输入
- 在需要时绑定工作区，让 AI 读取目录内资料

对 AI 来说，两者本质上都是当前任务的上下文材料，只是来源不同。

### 工作区是会话级上下文

工作区不应是全局概念，而应绑定到会话。

这样一条会话可以稳定围绕一批资料持续工作，不污染别的任务线程。

### 按需绑定

工作区绑定不是前置门槛。

只有当任务需要读取目录、批量处理文件或执行本地脚本时，AI 才在聊天中自然提示用户绑定工作区或补充资料。

### 面向任务的前台语言

前台不强调技术词汇，例如「文件系统权限」。

更自然的表达应是：

- 需要访问这次任务相关的资料
- 需要绑定一个工作区
- 需要读取这个目录中的内容

---

## 交付物设计

执行完成后，不应只停留在一条 assistant 回复。

每次关键结果都应沉淀成可引用的 artifact，例如：

- 已生成摘要
- 已导出文件
- 已输出脚本
- 已整理成表格

用户在会话里应能看见：

- 这次执行产出了什么
- 结果存放在哪里
- 是否可以继续基于它做下一步

内部可以把结果保存在应用私有目录中，但前台必须让用户有结果感知。

---

## 脚本生成策略

### 脚本的角色

脚本是一次 operation 的内部实现，不是产品主角。

它的使命是：

- 处理资料
- 批量操作
- 自动化重复步骤

### 语言策略

不建议让 AI 每次自由决定脚本语言。

平台应规定一个受控的内部自动化语言，优先考虑 `Python`，并由应用自己管理运行时。

原因：

- 更适合文本、表格、CSV、JSON、目录处理
- 更适合作为后续 skill 的稳定执行体
- 比多语言自由生成更容易审计、维护和复用

### 存放方式

脚本应保存在应用私有目录中，作为 operation 的执行资产，不直接暴露在主界面中。

---

## 从脚本到 skill 的沉淀

### 第一步：记录 operation

系统应记录每次脚本化执行的关键信息：

- 用户目标
- 输入材料类型
- 输出结果类型
- 使用过的步骤
- 脚本路径
- 执行结果

### 第二步：识别重复流程

重复识别不应只基于脚本文本相似，而应基于任务模式相似：

- 目标是否相近
- 输入输出结构是否相近
- 步骤结构是否相近

### 第三步：在聊天里提出建议

当系统判断这是重复流程时，在聊天中自然发一条 assistant 消息，例如：

> 我发现你最近多次做了类似整理流程，要不要把它保存成一个 skill，之后我可以直接复用。

### 第四步：升级为 skill

用户确认后，系统再创建 skill。

对普通用户，skill 是一个能力。
对高级用户，skill 是一个有路径和版本的资产。

### 第五步：通过自然语言继续修改

后续用户可以说：

- 以后这个流程默认先去重
- 这个 skill 也要支持 xlsx
- 这一步按客户分组再汇总

AI 应优先理解为修改已有 skill 的新版本，而不是重新发明一个新的 skill。

## 技术与产品的映射

基于当前仓库状态，下面这些方向已经有基础：

- `conversation/message` 顶层结构
- `agentMode` 切换
- 会话级 `working_dirs`
- 工具执行事件流
- 工具授权
- MCP 服务器管理
- 长文本 paste 持久化

因此，当前项目更适合沿着下面这条路径推进：

- 先把现有聊天客户端提升为 worker-first chat
- 再让 operation、artifact、skill 逐步长成内部一等对象

而不应现在就把产品重构成显式 workflow 平台。

---

## 风险与约束

### 风险一：产品仍停留在聊天壳

如果只强化消息流，而不把 operation 和 artifact 长出来，产品会停留在「偶尔会跑脚本的聊天工具」。

### 风险二：脚本语言和环境失控

如果允许模型自由选择语言或依赖宿主机环境，系统会很快变得不可控。

### 风险三：过早平台化

如果第一版就把 skill、版本、参数、builder 全部前台化，产品会变重，偏离当前用户心智。

### 风险四：自治失去信任

如果完全自主没有明确的高风险边界，用户不会长期信任系统。

---

## 分阶段落地建议

### Phase 1 - Worker-first Chat

- 保持首页和会话结构不变
- 补上简短计划消息
- 在聊天中展示执行步骤和动画
- 明确工作区按需绑定的交互
- 让结果以 artifact 形式出现

### Phase 2 - Operation 内部化

- 为脚本执行建立 operation 记录
- 关联输入、步骤、脚本和结果
- 为重复流程识别提供基础

### Phase 3 - Skill 沉淀

- 在聊天中自然建议保存 skill
- 建立 skill manifest、脚本路径和版本信息
- 支持自然语言修改 skill

---

## 当前结论

已确认的产品结论如下：

- 产品主形态是 Chat，不是任务看板或 IDE
- 默认模式是 Agent Chat，但允许退回普通聊天
- 每条会话只围绕一个持续任务线程展开
- AI 在执行前先给出简短计划
- 执行过程通过自然语言、步骤状态和动画展示
- 资料可以直接进入聊天，也可以通过工作区提供
- 工作区在需要时绑定，并按会话隔离
- AI 可以在后台写脚本完成任务，但代码默认隐藏
- 系统会识别重复流程，并在聊天中自然建议沉淀为 skill
- 用户可通过自然语言继续修改 skill，以适应同类工作流变化
